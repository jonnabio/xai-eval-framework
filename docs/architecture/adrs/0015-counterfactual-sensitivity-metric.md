# 15. Adoption of Counterfactual Sensitivity Metric

Date: 2025-12-27

## Status

Accepted

## Context

A key property of a faithful explanation is that it should highlight features that, if changed, would actually alter the model's prediction. "Counterfactual Explanations" specifically solve the optimization problem of finding the minimal feature changes required to flip a prediction.

We need a way to cross-validate feature attribution methods (SHAP/LIME) against counterfactual methods (DiCE) to ensure consistency. If SHAP says "Age" is the most important feature, but DiCE flips the prediction by changing only "Hours-per-week", there is a disagreement (low sensitivity).

## Decision

We will implement a **Counterfactual Sensitivity Metric** that measures the agreement between attribution-based explanations and counterfactual explanations generated by DiCE.

### Metric Definition

We define **Counterfactual Sensitivity** primarily as the **Recall** of the attribution's Top-K features with respect to the set of features modified in the counterfactual(s).

- **Modified Features ($F_{CF}$)**: The set of features that have different values in the counterfactual instance compared to the original.
- **Top-K Features ($F_{Attr}$)**: The top $K$ features ranked by the attribution method.

**Sensitivity Score**:
$$ \text{Sensitivity} = \frac{|F_{Attr} \cap F_{CF}|}{|F_{CF}|} $$

(Interpretation: What fraction of the features *actually required* to flip the prediction were successfully identified by the attribution method?)

We also track **Precision**:
$$ \text{Precision} = \frac{|F_{Attr} \cap F_{CF}|}{|F_{Attr}|} $$

### Consequences

### Positive
- **Cross-Method Validation**: Uses a completely different XAI approach (optimization-based) to validate attribution methods.
- **Actionability**: High sensitivity implies the explanation points to actionable changes.

### Negative
- **Computationally Expensive**: Generating counterfactuals (via DiCE) is significantly slower than simple inference, adding time to the evaluation loop.
- **DiCE Instability**: Counterfactual generation is not deterministic (unless constrained) and might find "adversarial" examples that don't reflect true feature importance, potentially penalizing a correct attribution method.
- **Data Compatibility**: DiCE often requires access to raw feature ranges/definitions, complicating the pipeline that uses preprocessed (One-Hot) data.

## Implementation
- **Wrapper**: `src.xai.dice_wrapper.DiCETabularWrapper`
- **Metric**: `src.metrics.sensitivity.CounterfactualSensivtyMetric`
- **Integration**: Integrated into `ExperimentRunner`, toggled via `metrics.counterfactual: true`.
